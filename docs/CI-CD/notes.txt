⚙️ 1. CI/CD og GitHub Actions
1. Hvilke tjek laver man i CI-pipeline?

Typiske CI (Continuous Integration) checks:

✔️ Build (kompiler kode)
✔️ Unit tests
✔️ Linting (kodekvalitet)
✔️ Security checks (SAST)
✔️ Dependency check (sårbarheder)
✔️ Packaging (JAR / Docker image)

👉 Formål:

“Sikre at kode ikke bryder main branch”

2. jobs vs steps i GitHub Actions
🧩 jobs:
Separate “kørsler”
Kan køre parallelt
Kører på hver sin runner
🪜 steps:
Sekventielle handlinger i ét job
Kører i samme miljø

👉 Kort sagt:

job = “maskine”
step = “handlinger på maskinen”
3. uses vs run
uses:
Genbruger actions (færdige plugins)

Eksempel:

uses: actions/checkout@v4
run:
Kører shell commands direkte

Eksempel:

run: mvn test
4. Hvornår er run nyttig?

Bruges til:

build commands
tests
scripts
Docker commands
deployment scripts

Eksempler:

run: mvn clean install
run: npm test
run: docker build -t app .
5. Caching i GitHub Actions + cache key

Caching = genbrug af dependencies

Eksempel:

Maven dependencies
Node modules
Cache key:
unik identifikator
ofte baseret på hash af dependencies

Eksempel:

key: ${{ runner.os }}-maven-${{ hashFiles('**/pom.xml') }}

👉 Hvis pom.xml ændrer sig → cache invalidates

6. Hash-pinning (vigtigt eksamenspunkt)

Hash-pinning betyder:

Du låser en action til en specifik version via commit hash

Eksempel:

❌ Uden pin:

uses: actions/checkout@v4

✔️ Med pin:

uses: actions/checkout@f43a2b91c0d...

👉 Fordel:

Sikkerhed
Ingen “supply chain changes”
Reproducerbare builds

9. Docker build + push i GitHub Actions

Typisk flow:

- name: Build Docker image
  run: docker build -t myapp .

- name: Login to registry
  run: echo $TOKEN | docker login ghcr.io -u USER --password-stdin

- name: Push image
  run: docker push myapp

👉 bruges til deployment i cloud pipelines

💻 DEMONSTRATIONER
1. Java CI pipeline (Maven)
name: Java CI

on: push

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4
      - name: Set up Java
        uses: actions/setup-java@v4
        with:
          java-version: '17'

      - name: Build and test
        run: mvn clean test

👉 Forklaring:

checkout → henter kode
setup-java → runtime
mvn test → CI check
2. Maven caching (meget vigtigt i eksamen)
- name: Cache Maven dependencies
  uses: actions/cache@v4
  with:
    path: ~/.m2
    key: ${{ runner.os }}-maven-${{ hashFiles('**/pom.xml') }}
    restore-keys: |
      ${{ runner.os }}-maven-

👉 Effekt:

hurtigere builds
genbrug dependencies
3. workflow_dispatch (manuel CI)
on:
  push:
  workflow_dispatch:

👉 Du kan nu:

trykke “Run workflow”
teste CI uden commit

Cloud connection:
build → test → docker → deploy
